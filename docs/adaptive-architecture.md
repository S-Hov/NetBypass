# Адаптивная архитектура NetBypass

## 1. Общая идея

NetBypass должен развиваться из редактора статического `hosts` в адаптивный
менеджер доступа к отдельным сервисам.

Программа не должна безусловно применять один метод ко всем сайтам. Сначала она
определяет, почему конкретный сервис недоступен, затем выбирает минимальную
подходящую стратегию:

```text
Сервис
  ↓
Диагностика сети
  ├── настоящий IP доступен → Direct + Encrypted DNS
  ├── DNS искажён → Encrypted DNS
  ├── требуется другой регион → SNI Relay
  ├── обнаружена фильтрация DPI → подключаемый Anti-DPI движок
  └── безопасного маршрута нет → Unavailable + объяснение причины
```

Основной принцип: использовать для каждого сервиса самый простой работающий
способ, не перенаправляя весь трафик компьютера.

NetBypass по-прежнему не должен позиционироваться как VPN:

- внешний IP не меняется для всего трафика;
- сетевой туннель не включается без необходимости;
- обрабатываются только выбранные домены;
- пользователь видит применённую стратегию и результат диагностики.

## 2. Источники архитектурных идей

Ни один проект не следует копировать целиком. NetBypass заимствует проверенные
подходы и реализует собственный управляющий слой.

### dns.malw.link

Источник: <https://github.com/ImMALWARE/dns.malw.link>

Используемые идеи:

- каталог доменов, сгруппированных по сервисам;
- DNS-перенаправление только выбранных доменов;
- использование SNI-прокси для обхода региональных ограничений;
- разделение DNS-сервера и проксирующего узла.

Что следует улучшить:

- не закреплять один IP релея в статическом `hosts`;
- проверять доступность узла перед применением;
- поддерживать несколько источников и резервных узлов;
- не считать наличие записей доказательством работоспособности.

### SniDust

Источник: <https://github.com/Seji64/SniDust>

Используемые идеи:

- развёртывание SmartDNS и SNI-релея на собственном VPS;
- список разрешённых клиентов;
- контейнерное развёртывание инфраструктуры;
- разделение обычного DNS-трафика и проксируемых сервисов.

Что важно для NetBypass:

- собственный сервер пользователя должен быть поддерживаемым вариантом;
- публичный релей нельзя оставлять полностью открытым;
- серверный шаблон должен ограничивать клиентов по IP, токену либо частной
  сети.

SniDust распространяется под GPL-3.0. Его исходный код нельзя переносить в
NetBypass без отдельного решения о совместимости лицензий.

### mosajjal/sniproxy

Источник: <https://github.com/mosajjal/sniproxy>

Используемые идеи:

- совмещение DNS и SNI-прокси в одном серверном компоненте;
- ACL для доменов и клиентов;
- поддержка UDP/TCP DNS, DoT, DoH и DoQ;
- GeoIP-правила;
- несколько исходящих IP;
- метрики и наблюдаемость;
- возможность использовать upstream SOCKS5.

Для первой серверной версии NetBypass этот проект следует рассматривать как
готовый внешний компонент, а не переписывать его возможности на C#.

### ameshkov/sniproxy

Источник: <https://github.com/ameshkov/sniproxy>

Используемые идеи:

- отдельные правила для перенаправления, прямого подключения, блокировки и
  forwarding;
- направление выбранных доменов через SOCKS5 или HTTP-прокси;
- SNI/Host-маршрутизация без расшифровки HTTPS.

Это основа подключаемых стратегий: один сервис может работать напрямую, другой
через SNI-релей, третий через пользовательский SOCKS5.

### SmartDNS

Источник: <https://github.com/pymumu/smartdns>

Используемые идеи:

- параллельный запрос нескольких DNS upstream;
- получение нескольких адресов одного домена;
- измерение доступности и задержки;
- выбор самого быстрого доступного адреса;
- резервирование при отказе DNS-сервера;
- DoH, DoT, DoQ и DNS через прокси.

NetBypass не должен использовать простой ICMP ping как единственный критерий.
Для HTTPS-сервисов необходимо проверять TCP, TLS с правильным SNI и, при
необходимости, HTTP-ответ.

### dnscrypt-proxy

Источник: <https://github.com/DNSCrypt/dnscrypt-proxy>

Используемые идеи:

- зашифрованные DNS-протоколы;
- измерение и балансировка resolver-ов;
- кеширование;
- автоматическое обновление доверенных списков;
- cloaking rules как динамическая замена `hosts`;
- логирование обычных и подозрительных DNS-запросов.

На Windows dnscrypt-proxy можно поддерживать как внешний DNS-движок. Собственная
реализация всех протоколов не нужна на раннем этапе.

### GoodbyeDPI

Источник: <https://github.com/ValdikSS/GoodbyeDPI>

Используемые идеи:

- отдельная стратегия для случаев DPI-фильтрации;
- перенаправление DNS для защиты от DNS poisoning;
- применение правил только к целевым доменам;
- набор диагностируемых стратегий вместо одного универсального режима.

GoodbyeDPI не следует встраивать в основной процесс NetBypass. Это должен быть
опциональный внешний движок с отдельным жизненным циклом и явным согласием
пользователя.

### zapret

Источник: <https://github.com/bol-van/zapret>

Используемые идеи:

- диагностика типа ограничения перед подбором стратегии;
- разные профили для разных провайдеров и протоколов;
- проверка DNS, TCP, TLS и HTTP отдельно;
- автоматизированный перебор допустимых методов.

Полный `zapret` ориентирован прежде всего на Linux/OpenWrt и не должен быть
основным Windows-движком NetBypass. Ценность для проекта — диагностическая
модель и профильный подход.

## 3. Целевая архитектура

```text
┌───────────────────────────┐
│        WPF Client         │
│ Главная / Сервисы / Диагн.│
└─────────────┬─────────────┘
              │
┌─────────────▼─────────────┐
│     NetBypass Engine      │
│ Profiles · Plans · State  │
└───┬─────────┬─────────┬───┘
    │         │         │
┌───▼───┐ ┌───▼────┐ ┌──▼──────────┐
│Probe  │ │Catalog │ │Strategy     │
│Engine │ │Manager │ │Providers    │
└───┬───┘ └───┬────┘ └──┬──────────┘
    │         │          │
    │         │     ┌────┼───────────────┐
    │         │     │    │               │
┌───▼───┐ ┌───▼──┐ ▼    ▼               ▼
│DoH/DoT│ │Signed│Hosts Local DNS   External Engine
│TCP/TLS│ │Feeds │Mode  Service     SNI/SOCKS/Anti-DPI
└───────┘ └──────┘
```

### WPF Client

Отвечает только за управление и отображение:

- состояние каждого сервиса;
- выбранная и реально применённая стратегия;
- результат последней диагностики;
- доступность внешних релеев;
- безопасное включение, отключение и восстановление.

GUI не должен самостоятельно выполнять сетевые проверки или записывать
`hosts`.

### NetBypass Engine

Центральный управляющий слой:

- загружает профили сервисов;
- запускает диагностику;
- получает кандидатов стратегий;
- строит `ConnectionPlan`;
- применяет план через провайдер стратегии;
- сохраняет только проверенное состояние;
- повторно проверяет результат после применения.

### Probe Engine

Диагностика должна быть многоступенчатой:

1. получить настоящие DNS-ответы через несколько DoH upstream;
2. получить ответы специализированных источников релеев;
3. проверить TCP-подключение;
4. выполнить TLS handshake с правильным SNI;
5. проверить сертификат;
6. отправить безопасный HTTP `HEAD` или другой профильный запрос;
7. классифицировать результат;
8. записать задержку и время проверки.

Один успешный TCP connect недостаточен. Ответ `403` может означать как рабочее
соединение с региональным запретом, так и Cloudflare challenge. Это должен
интерпретировать профиль сервиса.

### Catalog Manager

Объединяет несколько источников данных:

- встроенный каталог NetBypass;
- адаптер импорта dns.malw.link;
- пользовательские профили;
- подписанный каталог сообщества;
- пользовательские релей-серверы.

Все внешние данные проходят:

- проверку схемы;
- нормализацию доменов;
- проверку запрещённых локальных адресов;
- дедупликацию;
- проверку цифровой подписи;
- сохранение предыдущей рабочей версии;
- аудит изменений перед активацией.

### Strategy Providers

Каждая стратегия реализует единый контракт:

```csharp
public interface IAccessStrategy
{
    string Id { get; }
    Task<StrategyProbeResult> ProbeAsync(
        ServiceProfile service,
        NetworkContext network,
        CancellationToken cancellationToken);
    Task<ApplyResult> ApplyAsync(
        ConnectionPlan plan,
        CancellationToken cancellationToken);
    Task RemoveAsync(
        ConnectionPlan plan,
        CancellationToken cancellationToken);
}
```

Первые провайдеры:

- `DirectDohStrategy`;
- `AdaptiveHostsStrategy`;
- `SniRelayStrategy`;
- `UserProxyStrategy`.

Позднее:

- `LocalDnsStrategy`;
- `AntiDpiStrategy`.

## 4. Модель данных

### ServiceProfile

Статический профиль сервиса:

```yaml
schemaVersion: 1
id: openai
name: OpenAI
category: AI
domains:
  required:
    - chatgpt.com
    - auth.openai.com
    - cdn.oaistatic.com
  optional:
    - api.openai.com
strategies:
  - direct-doh
  - sni-relay
healthChecks:
  - host: chatgpt.com
    port: 443
    protocol: https
    expected:
      tlsCertificate: valid
      httpStatuses: [200, 301, 302, 403]
```

Профиль не должен содержать один постоянный IP. IP является результатом
динамического разрешения или конфигурацией конкретного релея.

### RelayEndpoint

```yaml
id: community-eu-1
address: 203.0.113.10
port: 443
protocol: sni
regions: [EU]
supportedServices: [openai, claude]
source: signed-community
expiresAt: 2026-07-01T00:00:00Z
```

Дополнительные поля:

- публичный ключ или идентификатор источника;
- результат последней проверки;
- задержка;
- история ошибок;
- допустимые клиенты;
- признак собственного сервера.

### ProbeResult

```csharp
public sealed record ProbeResult(
    ProbeStage Stage,
    ProbeStatus Status,
    TimeSpan? Latency,
    string? Address,
    string? ErrorCode,
    string? Message,
    DateTimeOffset CheckedAt);
```

Этапы:

- `Dns`;
- `Tcp`;
- `Tls`;
- `Http`;
- `Service`.

### ConnectionPlan

Результат принятого решения:

```csharp
public sealed record ConnectionPlan(
    string ServiceId,
    string StrategyId,
    IReadOnlyList<string> Domains,
    string? SelectedAddress,
    string SourceId,
    DateTimeOffset CreatedAt,
    DateTimeOffset ExpiresAt);
```

План должен быть воспроизводимым: при следующем запуске программа может
сравнить сохранённый план с фактической системной конфигурацией.

## 5. Алгоритм выбора стратегии

Для каждого выбранного сервиса:

1. загрузить профиль;
2. получить настоящие адреса через минимум два доверенных DoH resolver-а;
3. проверить прямое подключение;
4. если сервис доступен и профиль допускает direct — выбрать `DirectDoh`;
5. если DNS-ответы отличаются или обычный DNS искажён — выбрать
   `DirectDoh` с локальным DNS;
6. если соединение есть, но сервис сообщает региональное ограничение —
   проверить SNI-релеи;
7. проверить все допустимые релеи параллельно с ограничением конкурентности;
8. выбрать рабочий релей по результату TLS/HTTP и задержке;
9. если настроен пользовательский SOCKS5 — проверить его как отдельный
   кандидат;
10. запускать Anti-DPI диагностику только при признаках DPI и только после
    явного включения пользователем;
11. если кандидатов нет — установить `Unavailable`, не применяя заведомо
    нерабочую конфигурацию.

Приоритеты по умолчанию:

```text
Direct → Encrypted DNS → Own Relay → Trusted Relay → User Proxy → Anti-DPI
```

Пользователь может изменить порядок, но безопасный прямой маршрут всегда
предпочтительнее стороннего посредника.

## 6. Режимы применения

### Adaptive Hosts

Первый этап развития текущего приложения:

- движок выбирает проверенный IP;
- текущий `HostsFileService` записывает его в управляемый блок;
- план имеет срок жизни;
- при следующем запуске адрес проверяется заново;
- статус называется «Записи применены и проверены», а не просто «Активно».

Ограничение: статический `hosts` не может динамически переключиться при отказе
узла без повторного запуска приложения.

### Local DNS Service

Следующий основной режим:

- Windows-служба слушает `127.0.0.1:53`;
- обычные домены отправляются в DoH/DoT upstream;
- выбранные домены получают адрес согласно активному плану;
- кеш имеет короткий TTL для релейных записей;
- служба выполняет failover без открытия GUI;
- системный DNS временно переводится на localhost и корректно восстанавливается.

Это заменит постоянное редактирование `hosts`, но потребует фоновой службы.

### Own Relay

Серверный пакет для VPS:

- DNS/SNI-компонент разворачивается через Docker Compose;
- по умолчанию закрыт от посторонних;
- поддерживает allowlist клиентов;
- имеет health endpoint и метрики;
- обновляется отдельно от Windows-клиента;
- не расшифровывает HTTPS;
- разрешает только домены из профилей пользователя.

### Anti-DPI Plugin

Опциональный поздний этап:

- отдельный подписанный исполняемый компонент;
- выключен по умолчанию;
- запускается только для выбранных доменов;
- имеет собственные профили и журнал;
- конфликтует с VPN/TUN-драйверами — конфликт должен обнаруживаться заранее;
- удаляется и останавливается независимо от DNS/hosts-режима.

## 7. Состояния интерфейса

Глобальный статус должен отражать реальную работоспособность:

- `NotConfigured`;
- `Diagnosing`;
- `ActiveVerified`;
- `ActiveDegraded`;
- `ChangesPending`;
- `Unavailable`;
- `ConfigurationCorrupted`.

Для каждого сервиса показываются:

- выбранная стратегия;
- адрес или релей;
- время последней проверки;
- задержка;
- срок действия плана;
- понятная причина ошибки.

Примеры:

```text
OpenAI — SNI Relay — EU-1 — 48 ms — проверено 2 минуты назад
Twitch — Direct DoH — Cloudflare — 31 ms
Claude — недоступен — все 3 релея отклонены
```

## 8. Безопасность

### Подписанные каталоги

Удалённый каталог должен распространяться как:

```text
catalog.json
catalog.json.sig
```

Клиент содержит доверенные публичные ключи. Неподписанные автоматические
обновления не активируются. Пользовательский импорт разрешён только с явным
предупреждением.

### Защита от вредоносных назначений

Запрещаются:

- loopback;
- private и link-local диапазоны;
- multicast;
- unspecified адреса;
- домены `.localhost`;
- перенаправления системно важных доменов без явного подтверждения.

Релей проходит TLS-проверку для каждого целевого домена. Нельзя считать
открытый порт признаком доверенного релея.

### Минимизация доверия

- прямой маршрут предпочтительнее релея;
- собственный релей предпочтительнее публичного;
- содержимое HTTPS не расшифровывается;
- секреты пользовательских прокси хранятся через Windows DPAPI;
- диагностика и журналы не сохраняют URL, токены и содержимое запросов;
- удалённая телеметрия выключена по умолчанию.

### Лицензии

Перед использованием бинарника или исходного кода каждого проекта требуется
отдельная проверка лицензии.

Архитектурная идея сама по себе обычно не требует переноса лицензии, но:

- копирование кода GPL-компонента может потребовать GPL для производной работы;
- внешний процесс с отдельной поставкой проще изолировать юридически;
- лицензии, copyright notices и исходники зависимостей должны поставляться по
  их условиям.

До выбора окончательной лицензии NetBypass нельзя переносить фрагменты чужого
кода.

## 9. План реализации

### Этап 1. Dynamic profiles and probes

- заменить `.hosts` на версионированные `ServiceProfile`;
- добавить DoH-клиент;
- реализовать DNS/TCP/TLS/HTTP проверки;
- хранить результаты диагностики;
- отделить «применено» от «проверено»;
- не менять существующий режим записи `hosts`.

Критерий готовности: NetBypass не применяет IP, который не прошёл профильную
проверку.

### Этап 2. Adaptive Hosts

- добавить источники адресов и релеев;
- реализовать параллельный выбор кандидатов;
- добавить `ConnectionPlan`;
- кешировать последний рабочий план;
- добавить срок действия и повторную проверку;
- показывать стратегию и причину выбора.

Критерий готовности: при отказе основного релея приложение выбирает другой
рабочий адрес при следующей диагностике.

### Этап 3. Signed catalogs

- определить JSON Schema профилей и каталога;
- добавить подпись Ed25519;
- добавить обновление и rollback;
- реализовать адаптер импорта dns.malw.link;
- добавить просмотр различий перед применением.

Критерий готовности: повреждённое или неподписанное обновление не заменяет
последний рабочий каталог.

### Этап 4. Local DNS Service

- создать отдельную Windows-службу;
- добавить IPC между WPF и службой;
- реализовать локальный DNS-кеш;
- переключать системный DNS транзакционно;
- восстанавливать настройки после сбоя;
- реализовать автоматический failover релеев.

Критерий готовности: смена рабочего релея происходит без редактирования
`hosts` и без запуска GUI.

### Этап 5. Own Relay

- выбрать готовый SNI-прокси как серверную зависимость;
- подготовить Docker Compose;
- добавить ACL клиентов;
- добавить метрики и health checks;
- реализовать мастер подключения собственного VPS;
- документировать обновление и ротацию ключей.

Критерий готовности: пользователь может подключить собственный сервер без
ручного редактирования клиентских профилей.

### Этап 6. Optional Anti-DPI

- определить интерфейс внешнего движка;
- добавить проверку совместимости с VPN/TUN;
- реализовать ограничение по доменам;
- добавить безопасный запуск и остановку;
- подготовить отдельное пользовательское предупреждение.

Критерий готовности: Anti-DPI полностью отключается и не оставляет сетевые
фильтры после завершения.

## 10. Предлагаемая структура решения

```text
src/
├── NetBypass.App
├── NetBypass.Core
├── NetBypass.Diagnostics
├── NetBypass.Catalog
├── NetBypass.Strategies
├── NetBypass.Service
└── NetBypass.Contracts

server/
├── docker-compose.yml
├── relay/
└── monitoring/

schemas/
├── service-profile.schema.json
└── catalog.schema.json
```

Ответственность проектов:

- `Contracts` — общие DTO и интерфейсы;
- `Core` — оркестрация и планы;
- `Diagnostics` — сетевые проверки;
- `Catalog` — профили, подписи и обновления;
- `Strategies` — провайдеры маршрутов;
- `Service` — локальный DNS и фоновые задачи;
- `App` — пользовательский интерфейс.

## 11. Что реализовать первым

Ближайшая практическая итерация:

1. ввести `ServiceProfile` без удаления текущих модулей;
2. реализовать DoH-запросы к Cloudflare и Google;
3. добавить TCP/TLS-проверку с правильным SNI;
4. проверять статический IP модуля до записи;
5. отображать `ActiveVerified`, `ActiveDegraded` и `Unavailable`;
6. добавить экран диагностики с техническими подробностями;
7. покрыть проверки тестами через подменяемые сетевые интерфейсы.

Это даст немедленную пользу и создаст фундамент следующих этапов, не требуя
сразу устанавливать Windows-службу или поднимать собственный VPS.
